# Permission-Aware RAG: State of Practice (2025–2026)

## 1. Azure AI Search security trimming

Four patterns, in maturity order:

- **Security filter pattern (GA, canonical)**: store principal IDs in a filterable `Collection(Edm.String)` field (e.g. `group_ids`, `retrievable: false`), resolve the caller's groups at query time, and append `group_ids/any(g: search.in(g, 'g1,g2,...'))` to every query. `search.in` keeps this sub-second even with hundreds of IDs; it's pure string matching, no real authn — the app is trusted to build the filter. This is what most RAG apps (incl. the azure-search-openai-demo lineage) actually ship.
- **Native document-level ACL/RBAC (preview, 2025-05 → 2026-05-preview APIs)**: index stores POSIX-like ACL/RBAC permission metadata; caller's Entra token passed via `x-ms-query-source-authorization` header and the service itself trims results against user/group claims. Removes custom trimming code and nested-group resolution.
- **SharePoint M365 ACL ingestion** and **Purview sensitivity-label enforcement** (both preview) — indexers pull source ACLs/labels into the index; enforcement at query time against the Entra token.
- Key caveat across all: **permissions are enforced against index-synced metadata**, so there's a staleness lag between an ACL change at the source and the next indexer run/push update. Chunked indexes must project ACL fields onto every chunk row or chunk-level results won't be filtered.

Sources: [Document-level access overview](https://learn.microsoft.com/en-us/azure/search/search-document-level-access-overview), [Security filter pattern](https://learn.microsoft.com/en-us/azure/search/search-security-trimming-for-azure-search), [Entra-based doc-level security announcement](https://techcommunity.microsoft.com/blog/azure-ai-foundry-blog/announcing-enterprise-grade-microsoft-entra-based-document-level-security-in-azu/4418584)

## 2. Postgres + pgvector multi-tenant patterns

**Isolation spectrum** (Nile's framing): tenant-id column (cheapest, leak risk is app discipline) → RLS (defense-in-depth backstop) → schema/partition-per-tenant → database-per-tenant (max isolation, ops overhead; only for few high-value tenants).

- **RLS**: production benchmarks (50M rows / 10K tenants) show ~0.3ms policy overhead *if* composite indexes lead with `tenant_id`; missing that index is ~100x slower. RLS holds up under 500-connection concurrency. Widely recommended as a backstop even when the app already filters.
- **The pgvector-specific problem — post-filter recall**: HNSW walks the graph first, then applies the filter. A selective filter (`tenant_id = 42`, or a per-user ACL predicate) can return fewer than k results or garbage recall. pgvector **0.8.0 iterative index scans** (`hnsw.iterative_scan = relaxed_order`) fix over-filtering by scanning deeper, at CPU/tail-latency cost. Official guidance: **few distinct filter values → partial indexes; many values → partition by that column**.
- **One filtered index vs many small collections**: the Curator paper and Pinecone/Weaviate guidance frame it as *per-tenant index* (better recall, index tuned to the tenant's distribution, unfiltered fast queries, small blast radius, fast deletes — but memory/management overhead per index) vs *shared index + metadata filter* (memory-efficient, flexible ACLs, but distribution mismatch + runtime filter cost). Weaviate's answer is shard-per-tenant inside one collection; Postgres's equivalent is **partitioning by tenant (or per-project table/partition), each partition with its own HNSW index**. Practitioners pushing 100K+ collections report metadata/catalog overhead becomes the limiter — so "smaller databases" per project/case is favored when projects are query-scoped anyway (the legal pattern, §3), and small per-project sets can even skip ANN entirely: exact scan on a few thousand rows gives perfect recall.
- Blast radius: shared index = one corrupted/rebuilt index affects everyone; per-partition index = rebuilds, re-embeds, and deletes are scoped.

Sources: [Nile multi-tenant RAG](https://www.thenile.dev/blog/multi-tenant-rag), [Nile pgvector 0.8.0](https://www.thenile.dev/blog/pgvector-080), [pgvector 0.8.0 release](https://www.postgresql.org/about/news/pgvector-080-released-2952/), [AWS Aurora pgvector 0.8.0](https://aws.amazon.com/blogs/database/supercharging-vector-search-performance-and-relevance-with-pgvector-0-8-0-on-amazon-aurora-postgresql/), [pgEdge filtering docs](https://docs.pgedge.com/pgvector/v0-8-1/filtering/), [RLS multi-tenancy benchmarks](https://ricofritzsche.me/mastering-postgresql-row-level-security-rls-for-rock-solid-multi-tenancy/), [AWS RLS pattern](https://aws.amazon.com/blogs/database/multi-tenant-data-isolation-with-postgresql-row-level-security/), [Curator paper](https://arxiv.org/abs/2401.07119), [Pinecone multi-tenancy](https://www.pinecone.io/learn/series/vector-databases-in-production-for-busy-engineers/vector-database-multi-tenancy/), [Weaviate multi-tenancy architecture](https://weaviate.io/blog/weaviate-multi-tenancy-architecture-explained), [ParadeDB pgvector limitations](https://www.paradedb.com/learn/postgresql/pgvector-limitations)

## 3. Legal AI: per-matter scoping + ethical walls

- **Harvey Vault**: retrieval is **project-scoped by construction** — each Vault project is its own corpus (~1,000 files/project limit) with explicit sharing (view/edit/full-access per user or workspace link); "knowledge bases" are the workspace-wide variant. Retrieval never spans projects a user can't see. This is the "many small databases" pattern in production.
- **Harvey × Intapp (Feb 2026)**: Intapp Walls for AI policies **auto-sync into Harvey's access and sharing controls** across Assistant, Vault, and Workflows — walls are enforced at the document-retrieval layer, context layer, and output layer; the system **fails closed** (skips a doc it can't confirm is inside the authorized boundary) and logs every document access/context window for audit. The external conflicts system is the source of truth; the AI platform is a policy consumer.
- **Legora**: every query authenticated and scoped to authorized data; permissioning at **both matter and document levels**; integrates with the DMS (NetDocuments, SharePoint) so documents stay in the system of record and their ACLs are respected; ISO 27001 / SOC 2 / ISO 42001.
- Takeaway: legal AI converged on *scope-first retrieval* (pick the matter corpus, then search) + *external policy sync* rather than one giant filtered index.

Sources: [Harvey/Intapp announcement](https://www.harvey.ai/blog/intapp-partners-with-harvey-bringing-ethical-wall-enforcement-directly-into-the-platform), [Intapp press](https://www.intapp.com/news/intapp-harvey-partner/), [Legal IT Insider](https://legaltechnology.com/2026/02/23/harvey-partners-with-intapp-for-ethical-walls-enforcement/), [Harvey Vault sharing](https://help.harvey.ai/articles/share-a-vault), [Vault knowledge bases](https://help.harvey.ai/articles/vault-knowledge-bases), [Legora product](https://legora.com/product), [Legora + NetDocuments](https://legora.com/newsroom/legora-netdocuments-secure-seamless-ai-powered-legal-work), [Legora on Azure OpenAI](https://www.microsoft.com/en/customers/story/23171-legora-azure-openai)

## 4. Chunking + citations; hybrid search as default

- **Per-chunk provenance is table stakes**: every chunk row carries `source_document_id`, page range / section title / char span, chunk ordinal, and a content hash; chunk boundaries aligned to structure (headings/pages) where possible because static page boundaries make citations trivial. Answers cite by mapping the retrieved chunk back to doc + page/span.
- Chunk-size evidence is mixed (recursive ~512-token splitting beat semantic chunking in one 2026 benchmark; structure-aware won elsewhere) — the stable advice is structure-aware boundaries + rich metadata, not exotic chunkers.
- **Hybrid (BM25/full-text + vector) fused with RRF is the 2025–26 default**, because ~1/3 of real queries contain exact identifiers embeddings miss. In Postgres: `tsvector` (or ParadeDB pg_search for true BM25) + pgvector, two ranked lists merged by RRF in one SQL function; one case study went 62% → 84% retrieval precision adding lexical+RRF. RRF is preferred over score normalization because BM25 and cosine scores aren't on comparable scales. Critically for you: **permission/tenant filters must be applied inside both arms of the hybrid query**, not after fusion.

Sources: [ParadeDB hybrid search manual](https://www.paradedb.com/blog/hybrid-search-in-postgresql-the-missing-manual), [pgvector + FTS + RRF walkthrough](https://dev.to/lpossamai/building-hybrid-search-for-rag-combining-pgvector-and-full-text-search-with-reciprocal-rank-fusion-6nk), [pg_trgm + pgvector hybrid](https://callsphere.ai/blog/vw7h-pg-trgm-pgvector-hybrid-retrieval-2026), [Redis on BM25 precision layer](https://redis.io/blog/full-text-search-for-rag-the-precision-layer/), [Firecrawl chunking strategies](https://www.firecrawl.dev/blog/best-chunking-strategies-rag), [NVIDIA chunking study](https://developer.nvidia.com/blog/finding-the-best-chunking-strategy-for-accurate-ai-responses/), [AlloyDB hybrid search docs](https://docs.cloud.google.com/alloydb/docs/ai/run-hybrid-vector-similarity-search)

## 5. Swappable embedding seam

- .NET has a first-party answer: **`Microsoft.Extensions.AI.IEmbeddingGenerator<string, Embedding<float>>`** (GA May 2025, with `Microsoft.Extensions.VectorData`) — app code depends on the interface; provider swap (OpenAI ↔ Azure OpenAI ↔ Ollama ↔ local) is one DI registration line.
- But vectors from different models are **not comparable** ("index drift") — so state of practice is: store `embedding_model`, version, and dimension **per chunk row**; migrations are full re-embed, lazy re-embed, or **dual-index with alias swap** (build new index in parallel, eval, atomically switch, instant rollback). `pgvector`'s typed `vector(n)` column means a dimension change = new column/table anyway — plan for it.

Sources: [Microsoft.Extensions.AI docs](https://learn.microsoft.com/en-us/dotnet/ai/microsoft-extensions-ai), [MEAI + VectorData GA](https://devblogs.microsoft.com/dotnet/ai-vector-data-dotnet-extensions-ga/), [RAG with .NET building blocks](https://nikiforovall.blog/dotnet/ai/2026/02/28/building-rag-with-dotnet-ai-building-blocks.html), [Embedding versioning & index drift](https://tianpan.co/blog/2026-04-09-embedding-models-production-versioning-index-drift), [Zilliz on embedding versioning](https://zilliz.com/ai-faq/how-do-i-handle-versioning-of-embedding-models-in-production), [Mixpeek embedding portability](https://mixpeek.com/guides/embedding-portability-versioning)

## Recommendations for Plenipo (.NET + Postgres, existing tenant filters / dotted permissions / ACL seam / job runner)

1. **Chunk table shape**: `chunks(id, tenant_id, collection_id, document_id, ordinal, text, tsv tsvector, embedding vector(n), embedding_model, page_from, page_to, span_start, span_end, content_hash, acl_snapshot…)`. Reuse your tenant global query filter on it; add **RLS as a backstop** with `tenant_id`-leading composite indexes — matches how you'd defend the raw SQL that hybrid search inevitably needs.
2. **Adopt the Azure-style trimming model on Postgres**: denormalize *coarse* access metadata onto chunks (collection/project id + your resource-ACL principal list), filter at query time inside the SQL, and re-check the per-resource ACL seam (owner/editor/viewer) on the final top-k before generation — a cheap "fail-closed" post-check exactly like Harvey's. Treat denormalized ACLs as a cache with a staleness contract; your background-job runner (captured-permission execution is a perfect fit) re-syncs ACL changes to chunk rows.
3. **Collections, not one big index**: model retrieval as **project/case-scoped collections** (Harvey Vault pattern) — a `collection_id` partition key, list-partitioned tables or per-collection partial indexes for big tenants, plain filtered queries for small ones. Small collections (<~10–50K rows) can use exact scan (perfect recall, zero index maintenance); build HNSW per partition only when needed. This sidesteps pgvector's filtered-recall problem, keeps blast radius (re-embeds, deletes, index rebuilds) per-project, and matches how legal/professional users think. Enable `hnsw.iterative_scan = relaxed_order` (pgvector ≥0.8.0) wherever you do filter a shared index.
4. **Hybrid by default**: one SQL function doing pgvector ANN + `tsvector` ranking (or ParadeDB pg_search if you want real BM25), fused with RRF, with tenant + collection + ACL predicates in **both** arms. Return `document_id + page/span` with every hit so answers cite; map citations through your tenant file store for click-through.
5. **Permission model mapping**: gate *collection* access with a dotted permission (e.g. `rag.collection.{id}.query` or your wildcard scheme) and *document* visibility with the existing owner/editor/viewer ACL — two layers, mirroring Intapp-wall (coarse policy) + DMS ACL (fine) in legal AI. Fail closed when ACL metadata is missing.
6. **Embedding seam**: depend on `IEmbeddingGenerator<string, Embedding<float>>` everywhere; store `embedding_model` + dimension per chunk; make re-embedding a background job per collection (your runner again) with a dual-column/dual-index swap for zero-downtime model migration.
